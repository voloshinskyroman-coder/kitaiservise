Архитектура продукта: AI-калькулятор доставки грузов из Китая

Главная идея

Мы создаем не обычный калькулятор доставки и не линейный квиз.

Мы создаем интеллектуальную систему, которая во время общения с пользователем постепенно строит цифровую модель его перевозки (Shipment Model).

Каждый ответ пользователя обновляет эту модель.

После каждого ответа система автоматически:

* обновляет параметры перевозки;
* пересчитывает ориентировочную стоимость;
* пересчитывает сроки доставки;
* пересчитывает точность расчета;
* пересчитывает качество лида (скоринг);
* принимает решение, какой вопрос задать следующим.

В конце ничего не рассчитывается заново.

К финальному экрану вся модель уже полностью собрана.

⸻

Главная модель системы

В центре всей архитектуры находится единый объект Shipment.

Он хранит абсолютно всю информацию о перевозке.

Shipment включает:

* цель обращения;
* тип доставки;
* категорию товара;
* описание товара;
* город отправления;
* город назначения;
* поставщика;
* статус поставщика;
* статус оплаты;
* стоимость товара;
* вес;
* объем;
* количество мест;
* размеры коробок (если объем неизвестен);
* дополнительные услуги;
* документы;
* предполагаемый маршрут;
* ориентировочный срок доставки;
* ориентировочную стоимость;
* диапазон стоимости;
* точность расчета;
* внутренний скоринг;
* комментарии системы.

Все остальные модули работают только с этим объектом.

⸻

Архитектура движков

Система состоит из нескольких независимых движков.

Quiz Engine

Отвечает только за отображение вопросов.

Он ничего не рассчитывает.

Его задача — показать следующий вопрос.

⸻

Decision Engine

Самый важный модуль.

Именно он определяет дальнейшую ветку сценария.

Каждый следующий вопрос выбирается на основании уже известных данных.

Если пользователь опытный, вопросов становится меньше.

Если пользователь новичок, приложение задает дополнительные уточняющие вопросы.

В результате каждый пользователь проходит собственный сценарий.

⸻

Shipment Engine

После каждого ответа обновляет объект Shipment.

Если пользователь указал вес, Shipment получает вес.

Если пользователь указал стоимость товара, Shipment получает стоимость.

Если пользователь выбрал белую доставку, Shipment получает соответствующий режим расчета.

⸻

Pricing Engine

Работает после каждого изменения Shipment.

После любого нового ответа полностью пересчитывает стоимость.

Стоимость никогда не считается один раз в конце.

Она постоянно обновляется.

При расчете используются:

* маршрут;
* вес;
* объем;
* категория товара;
* коэффициенты категории;
* дополнительные услуги;
* внутренние тарифы;
* стоимость оформления;
* стоимость доставки по Китаю;
* стоимость доставки по России;
* белая доставка (если выбрана);
* карго (если выбрано).

Результатом является диапазон стоимости, а не одна цифра.

⸻

White Delivery Engine

Работает только тогда, когда пользователь выбрал белую доставку.

Если выбрана карго-доставка, этот модуль полностью отключается.

Если включена белая доставка, модуль начинает учитывать:

* стоимость товара;
* примерную пошлину;
* НДС;
* стоимость оформления;
* брокерские услуги;
* необходимость ручной проверки.

⸻

Lead Scoring Engine

После каждого ответа оценивает качество клиента.

Скоринг никогда не считается только в конце.

Он постоянно изменяется.

Повышают скоринг:

* пользователь знает товар;
* знает поставщика;
* знает стоимость;
* знает вес;
* знает объем;
* товар уже оплачен;
* груз готов;
* понимает разницу между карго и белой доставкой;
* имеются документы.

Понижают скоринг:

* отвечает “не знаю”;
* не найден поставщик;
* неизвестен вес;
* неизвестен объем;
* неизвестна стоимость;
* просто интересуется.

На основании скоринга определяется:

горячий клиент;

теплый клиент;

холодный клиент.

Пользователь этого статуса не видит.

⸻

Validation Engine

Постоянно проверяет полноту данных.

Например:

если неизвестен вес,

точность расчета автоматически снижается.

Если неизвестны сразу несколько ключевых параметров,

система понимает, что точный расчет невозможен.

⸻

Recommendation Engine

После каждого изменения Shipment принимает решение, что можно показать пользователю.

Если данных достаточно,

показывается подробный расчет.

Если данных недостаточно,

показывается широкий диапазон стоимости и объясняется, каких данных не хватает.

⸻

Logistic Engine

После завершения квиза автоматически формирует карточку для логиста.

Логист должен получить полностью готовую заявку без необходимости повторно задавать клиенту те же вопросы.

⸻

Дерево сценариев

Все приложение строится вокруг дерева решений.

Первый вопрос определяет цель пользователя.

Далее система самостоятельно выбирает нужную ветку.

Основные сценарии:

1. Уже купил товар.
2. Только ищет товар.
3. Нужен выкуп.
4. Нужен перевод денег.
5. Белая доставка.
6. Карго.
7. Переход с карго на белую доставку.
8. Просто интересуется стоимостью.

Каждый сценарий имеет собственный набор последующих вопросов.

Никогда нельзя показывать пользователю длинный одинаковый квиз.

⸻

Логика расчета

Расчет начинается не в конце приложения.

Расчет начинается после первого ответа пользователя.

Каждый новый ответ запускает повторный пересчет всей модели.

Например:

после выбора города определяется маршрут;

после выбора категории определяется коэффициент товара;

после ввода веса определяется стоимость по весу;

после ввода объема определяется стоимость по объему;

после выбора дополнительных услуг стоимость снова обновляется;

после выбора белой доставки подключается модуль таможенного расчета.

В результате стоимость постоянно уточняется.

⸻

Логика вопросов

Каждый следующий вопрос должен иметь конкретную цель.

Он должен либо:

* повысить точность расчета;
* определить следующую ветку сценария;
* повысить качество лида;
* уточнить стоимость;
* определить необходимые услуги.

Если вопрос не влияет ни на одно из этих решений, его не должно быть в приложении.

⸻

Финальный экран

В конце пользователь получает:

* ориентировочную стоимость;
* диапазон стоимости;
* предполагаемый срок доставки;
* маршрут;
* выбранные услуги;
* информацию о точности расчета;
* предложение получить окончательный расчет от логиста.

⸻

Финальный результат для логиста

Логист получает структурированную карточку клиента.

Карточка содержит:

* всю модель Shipment;
* рассчитанную стоимость;
* диапазон стоимости;
* срок доставки;
* список выбранных услуг;
* недостающие данные;
* комментарии системы;
* внутренний скоринг;
* статус клиента.

Логист должен иметь возможность открыть карточку и сразу начать работу без повторного интервью.

⸻

Главные принципы проекта

1. Это не квиз, а интеллектуальная система диагностики перевозки.
2. Все строится вокруг объекта Shipment.
3. Все расчеты происходят после каждого ответа пользователя.
4. Каждый следующий вопрос зависит от предыдущего ответа.
5. Пользователь никогда не проходит одинаковый сценарий.
6. Главная задача приложения — собрать максимально качественную заявку для логиста.
7. Главный KPI продукта — точность предварительного расчета, качество квалификации лида и сокращение времени работы логиста с новой заявкой.
8. Архитектура должна быть модульной. Любой движок (расчет, скоринг, маршрутизация, рекомендации) должен развиваться независимо без изменения остальных частей системы.
