Контекст проекта

Мы создаём Telegram Mini App для расчёта стоимости доставки товаров из Китая (Taobao/1688/Pinduoduo и т.п.) под ключ.

Главная цель продукта:
получать тёплые и квалифицированные лиды на доставку и закрывать их через менеджеров.

Это продукт для обычного покупателя, а не для профессиональных байеров/логистов.

Он не должен разбираться в терминах логистики (карго, объёмный вес, растаможка и т.д.) — всё должно объясняться простым языком.

Главный user flow

1. Пользователь приходит из:

* Telegram
* Яндекс.Директ
* Instagram
* сайта / рекомендаций

2. Открывает Telegram Mini App
3. Вводит данные для расчёта:

* вес и/или габариты товара
* категория товара (одежда, техника, мебель, косметика и т.д.)
* откуда везём (город/маркетплейс в Китае, если нужно)
* количество мест
* нужна ли выкупка товара (или уже куплен)

4. Система считает:

* стоимость доставки
* примерные сроки
* (опционально) стоимость выкупа, если нужен агент

Логика калькулятора:
базовая ставка за кг/куб + коэффициенты по категории товара (хрупкое, техника, брендовое и т.д.) + срочность/тип доставки (авиа/жд/авто).

Пользователю формула не показывается.

Только:
итоговая цена / диапазон + срок.

5. Пользователь оставляет заявку (номер, что везём, откуда)
6. Заявка попадает в CRM/Supabase
7. Менеджер связывается с клиентом вручную или через бота

Роль менеджеров и холодных рассылок

Используем тот же движок outreach, что уже работает в проекте Vellar Home (`bots/outreach`):

* Telethon-аккаунты, прокси-пул, QR/2FA авторизация — переиспользуем инфраструктуру, не изобретаем заново
* Логика рассылок, обработки ответов и CRM-синка адаптируется под лиды KitaiService (другая база/таблицы, другие офферы и скрипты сообщений)
* Аккаунты, прокси и сессии для KitaiService — отдельные от Vellar, не смешивать пулы
* Supabase проект для KitaiService уже существует и отделён от Vellar — использовать его, credentials хранить вне git (например в `.env`, не в CLAUDE.md/коде)

Frontend:

Tailwind + premium clean UI, простой и быстрый калькулятор (минимум полей)

Backend:
Node.js / Next.js server actions

UX требования

Минимум friction — 1 экран с расчётом, максимум 30 секунд до результата.

Не перегружать логистическими терминами.

Человек должен ощущать:
"я за 30 секунд узнал сколько будет стоить доставка".

Это не форма для заполнения — это быстрый калькулятор.
